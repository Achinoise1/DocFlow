# DocFlow 渐进式重构方案

> 基于 V0.1.0 架构分析报告 + 模块重构规划制定。
> 原则：每一步独立可验证，不影响已有功能，可随时暂停。

---

## 总览

| Step | 名称 | 类型 | 风险 |
|------|------|------|------|
| 1 | 新建转换类型注册表 | 纯新增文件 | 🟢 零风险 |
| 2 | main_window 切换读取注册表 | 改现有文件 | 🟢 低风险 |
| 3 | file_utils 清理冗余常量 | 改现有文件 | 🟢 低风险 |
| 4 | 新建转换路由调度器 | 纯新增文件 | 🟢 零风险 |
| 5 | task_manager 接入调度器 | 改现有文件 | 🟡 中等风险 |
| 6 | 新建主题管理器 | 纯新增文件 | 🟢 零风险 |
| 7 | main_window 接入主题管理器 | 改现有文件 | 🟢 低风险 |

**建议节奏**：每完成一个 Step，运行一次程序做功能冒烟测试，确认后再进行下一步。

## Step 1｜新建转换类型注册表

### 目标
建立 `CONVERSION_REGISTRY` 作为全局唯一的转换类型知识库，替代当前散落在三处的重复数据。

### 修改文件
| 文件 | 操作 |
|------|------|
| `core/conversion_registry.py` | **新建** |

### 要做的事
新建文件，定义 `CONVERSION_REGISTRY`，为一个有序列表，每个条目是字典，包含以下字段：

```
id          str   转换类型唯一标识符，如 'word_to_pdf'
label       str   下拉框显示文字，如 'Word → PDF'
input_exts  list  接受的输入扩展名，如 ['.doc', '.docx']
output_ext  str   输出扩展名，如 '.pdf'
hint_text   str   拖拽区提示文字
tab         str   所属 Tab：'doc' / 'pdf' / 'image'
is_batch    bool  是否为批量合并模式（多文件合一）
```

同时提供两个辅助查询函数：
- `get_by_id(conversion_type_id)` → 返回对应条目 dict，找不到返回 None
- `get_by_tab(tab_name)` → 返回该 tab 下所有条目的列表

### 验证方法
```
python -c "from core.conversion_registry import CONVERSION_REGISTRY, get_by_id; print(get_by_id('word_to_pdf'))"
```
能正常打印条目即为成功。此步不触动任何现有文件，完全零风险。

## Step 2｜main_window 切换读取注册表

### 目标
删除 `main_window.py` 中硬编码的 `ext_map` 字典和 `_HINT_MAP` 类变量，改为从 Step 1 的注册表读取。

### 修改文件
| 文件 | 操作 |
|------|------|
| `ui/main_window.py` | **修改** |

### 要做的事

**改动 1：`_get_accepted_extensions()` 方法**

当前是一个本地 `ext_map` 字典硬编码了每个 conversion_type 对应的扩展名列表。
改为：从 `CONVERSION_REGISTRY` 的 `get_by_id(conversion_type)` 读取 `input_exts` 字段。

```
# 改前（伪示意）
ext_map = { 'word_to_pdf': ['.doc', '.docx'], ... }
return ext_map.get(conversion_type, [])

# 改后
entry = get_by_id(conversion_type)
return entry['input_exts'] if entry else []
```

**改动 2：`_HINT_MAP` 类变量 + `_on_conversion_type_changed()`**

当前 `_HINT_MAP` 是硬编码在类变量里的字典。
改为：`_on_conversion_type_changed()` 内直接调用 `get_by_id(conversion_type)['hint_text']`，删除 `_HINT_MAP`。

**改动 3：`_create_tab_page()` 中的 combo 条目**

当前三个 `if tab_type == 'pdf' / 'image' / 'doc'` 分支里手写了 `combo.addItem(...)` 条目。
改为：从 `get_by_tab(tab_type)` 读取，循环 `combo.addItem(entry['label'], entry['id'])`。

> ⚠️ 注意：`to_pdf` 这个 `conversion_type`（文档 Tab 的兼容模式）在注册表里不存在，需要确认 `_do_convert` 里的 `to_pdf` 分支是否还在使用，若已被 `word_to_pdf` / `ppt_to_pdf` 覆盖，则可以一并清理。

### 验证方法
启动程序，逐一切换三个 Tab 和每个 Tab 的下拉项：
- 拖拽区提示文字随下拉选择正确变化
- 拖入不匹配文件时提示被拒绝
- 下拉菜单条目与原来一致

## Step 3｜file_utils 清理冗余常量

### 目标
删除 `file_utils.py` 中的 `CONVERSION_MAP` 和 `SUPPORTED_EXTENSIONS`，改用注册表作为唯一来源。

### 修改文件
| 文件 | 操作 |
|------|------|
| `utils/file_utils.py` | **修改** |

### 要做的事

**改动 1：删除 `SUPPORTED_EXTENSIONS` 字典**

改为：`get_file_type()` 从 `CONVERSION_REGISTRY` 所有条目的 `input_exts` 反向构建 ext→type 的映射，或者直接在注册表里给每个条目加一个 `file_type` 标记字段（`'word'` / `'pdf'` / `'image'` 等），在 Step 1 时补充进去。

**改动 2：删除 `CONVERSION_MAP` 字典**

此字典仅用于描述输入/输出扩展名的对应关系，Step 1 注册表已覆盖此信息，直接删除。

**改动 3：`is_supported_file()` 维持可用**

改为查询注册表：取所有条目的 `input_exts` 展平为集合，判断文件扩展名是否在集合内。

> ⚠️ 注意：删除前确认 `CONVERSION_MAP` 在项目里是否有其他使用方（grep 一下），当前看来仅在 `file_utils.py` 内部定义但没有被其他文件 import。

### 验证方法
启动程序，拖入各类型文件，确认：
- 支持的文件能被正常识别（图标、类型显示正确）
- 不支持的文件被拒绝
- 文件大小显示正常（与 `CONVERSION_MAP` 无关，确认函数未被误删）

## Step 4｜新建转换路由调度器

### 目标
新建 `core/converter/dispatcher.py`，将 `task_manager.py` 中的 `if-elif` 路由链提取为独立函数。

### 修改文件
| 文件 | 操作 |
|------|------|
| `core/converter/dispatcher.py` | **新建** |

### 要做的事
新建文件，提供两个函数：

**函数 1：`dispatch(conversion_type, input_path, output_dir, options) -> str`**

将 `ConvertTask._do_convert()` 和 `_convert_to_pdf()` 的全部 `if-elif` 逻辑原样搬过来，不做任何修改。直接 import 所需的三个 converter 模块。

**函数 2：`dispatch_batch(conversion_type, image_paths, output_path) -> str`**

将 `BatchImageTask.run()` 中的 `if-elif` 逻辑原样搬过来。

> 此步只是"复制"，不删除原来的代码。原有 task_manager.py 保持不变、原功能完全正常。

### 验证方法
```
python -c "from core.converter.dispatcher import dispatch, dispatch_batch; print('ok')"
```
能正常 import 即可（不需要实际调用转换，避免依赖 Office 环境）。

## Step 5｜task_manager 接入调度器

### 目标
将 `task_manager.py` 中 `ConvertTask._do_convert()` 和 `BatchImageTask.run()` 的路由逻辑
替换为对 `dispatcher` 的调用，删除已迁移的冗余代码。

### 修改文件
| 文件 | 操作 |
|------|------|
| `core/task_manager.py` | **修改** |

### 要做的事

**改动 1：`ConvertTask._do_convert()` 方法体**

原来的 30 行 `if-elif` 链，替换为：
```
# 改后（伪示意）
from core.converter.dispatcher import dispatch
return dispatch(self.conversion_type, self.input_path, self.output_dir, self.options)
```
删除 `_convert_to_pdf()` 方法（已迁入 dispatcher）。

**改动 2：`BatchImageTask.run()` 中的路由逻辑**

原来的 `if self.conversion_type == 'images_to_pdf' / 'images_to_word'` 分支，替换为：
```
from core.converter.dispatcher import dispatch_batch
dispatch_batch(self.conversion_type, self.image_paths, self.output_path)
```

**改动 3：清理文件顶部的冗余 import**

删除原有模块顶部 `from core.converter.office_converter import word_to_pdf, ppt_to_pdf`（已通过 dispatcher 调用），以及方法内重复的 `from core.converter.office_converter import ...`。

> ⚠️ 这是整个重构中实际运行路径变化最大的一步，务必逐条转换类型做功能测试。

### 验证方法
依次测试每一种转换类型：
- Word → PDF
- PPT → PDF
- PDF → Word
- PDF → PPT
- PDF → 图片
- 图片 → PDF（批量合并）
- 图片 → Word（批量合并）

## Step 6｜新建主题管理器

### 目标
新建 `ui/theme_manager.py`，将主题逻辑封装为独立类，当前 main_window 代码不变。

### 修改文件
| 文件 | 操作 |
|------|------|
| `ui/theme_manager.py` | **新建** |

### 要做的事
新建文件，提供 `ThemeManager` 类，包含：

- `__init__(settings: QSettings)`：接收已有的 QSettings 实例
- `load_saved_theme() -> str`：从 QSettings 读取上次保存的主题名，默认 `'light'`
- `apply(theme_name: str)`：加载对应 QSS 文件并应用到 `QApplication`（逻辑从 `_apply_theme()` 原样搬过来，含冻结包路径修正逻辑）
- `toggle(current_theme: str) -> str`：返回切换后的主题名
- `get_button_icon(theme_name: str) -> str`：返回对应图标字符（`'🌙'` / `'☀️'`）

> 此步只是新建，不改 main_window.py，不影响任何功能。

### 验证方法
```
python -c "from ui.theme_manager import ThemeManager; print('ok')"
```

## Step 7｜main_window 接入主题管理器

### 目标
main_window 的 `_apply_theme()`、`_toggle_theme()`、`_update_theme_button()` 方法体改为调用 `ThemeManager`，删除其中的 QSS 读取逻辑。

### 修改文件
| 文件 | 操作 |
|------|------|
| `ui/main_window.py` | **修改** |

### 要做的事

**改动 1：`__init__` 中初始化 ThemeManager**

在 `self._settings = QSettings(...)` 之后立即创建：
```
self._theme_manager = ThemeManager(self._settings)
self._current_theme = self._theme_manager.load_saved_theme()
```

**改动 2：`_apply_theme()` 方法体**

原来约 15 行的 QSS 加载逻辑，替换为：
```
self._theme_manager.apply(theme_name)
self._current_theme = theme_name
self._update_theme_button()
```

**改动 3：`_update_theme_button()` 方法体**

替换为：
```
icon = self._theme_manager.get_button_icon(self._current_theme)
self.theme_btn.setText(icon)
```
（ToolTip 文字可以保留在 main_window 或移入 ThemeManager，二选一）

### 验证方法
启动程序：
- 上次选择的主题能正确恢复
- 点击主题按钮能切换深色/浅色
- 重启后主题记忆正确

## Step 8｜新建文件列表组件（骨架）

### 目标
新建 `ui/widgets/file_list_widget.py`，实现文件列表组件的完整逻辑，当前 main_window 不接入。

### 修改文件
| 文件 | 操作 |
|------|------|
| `ui/widgets/file_list_widget.py` | **新建** |

### 要做的事
新建 `FileListWidget(QWidget)` 类，将 `main_window.py` 中以下逻辑原样迁入：

| 迁入的内容 | 来源 |
|----------|------|
| `_file_items` 字典（内部状态） | `MainWindow._file_items` |
| `add_files(file_paths, accepted_exts)` | `MainWindow._add_files()` |
| `remove_file(file_path)` | `MainWindow._remove_file()` |
| `clear_files()` | `MainWindow._clear_files()` |
| `collect_files_from_dir(dir_path, accepted_exts)` | `MainWindow._collect_files_from_dir()` |
| `update_task_state(task_id, state, percent)` | 合并自 `_on_task_started/progress/finished` 中的 widget 更新部分 |

对外暴露信号：
- `file_count_changed = Signal(int)`：文件数量变化时发出
- `file_removed = Signal(str)`：单个文件被移除时发出

对外暴露属性/方法：
- `file_paths` → `list`：返回当前所有文件路径
- `task_id_map` → `dict`：返回 `{task_id: [file_path, ...]}` 的映射（供 main_window 查询）

> 此步主窗口仍使用旧代码，FileListWidget 只是新建但不接入，零风险。可单独肉眼 review 逻辑是否完整。

### 验证方法
```
python -c "from ui.widgets.file_list_widget import FileListWidget; print('ok')"
```

## Step 9｜main_window 接入文件列表组件

### 目标
将 main_window 中的文件列表管理逻辑切换为使用 `FileListWidget`，删除旧代码。

### 修改文件
| 文件 | 操作 |
|------|------|
| `ui/main_window.py` | **修改（改动最大）** |

### 要做的事

**改动 1：布局中实例化 FileListWidget**

原来 `_init_ui()` 里手动创建 `scroll_area`、`file_list_container`、`file_list_layout` 的部分，替换为：
```
self.file_list_widget = FileListWidget()
self.file_list_widget.file_count_changed.connect(self._update_file_count_label)
file_list_layout.addWidget(self.file_list_widget)
```

**改动 2：删除 main_window 中已迁入组件的方法**

- `_file_items` 字典 → 删除，使用 `self.file_list_widget.file_paths`
- `_add_files()` → 删除，改为 `self.file_list_widget.add_files(...)`
- `_remove_file()` → 删除
- `_clear_files()` → 改为 `self.file_list_widget.clear_files()`
- `_collect_files_from_dir()` → 删除

**改动 3：任务回调中的 widget 更新**

`_on_task_started()`、`_on_task_progress()`、`_on_task_finished()` 里遍历 `_file_items` 找 widget 的逻辑，改为调用 `self.file_list_widget.update_task_state(task_id, ...)`。

**改动 4：`_start_conversion()` 中读取文件列表**

`self._file_items.keys()` 改为 `self.file_list_widget.file_paths`。

> ⚠️ 这是整个重构中 UI 变化最大的一步，改完后务必全流程测试：
> 拖入文件 → 添加 → 删除 → 清空 → 转换 → 批量合并 → 进度更新 → 完成自动打开

### 验证方法
全流程测试：
- 拖拽文件进入列表
- 点击 × 移除单个文件
- 点击"清空列表"
- 切换转换类型自动剔除不兼容文件
- 转换进度条正常更新
- 转换完成状态显示正确（✓ 完成 / ✕ 失败）
- 取消转换后状态重置

## 各 Step 之间的依赖关系

```
Step 1（注册表）
  ├── Step 2（main_window 读注册表）
  └── Step 3（file_utils 清理）

Step 4（dispatcher）
  └── Step 5（task_manager 接入）

Step 6（ThemeManager）
  └── Step 7（main_window 接入主题）

Step 8（FileListWidget 骨架）
  └── Step 9（main_window 接入列表组件）
        ↑（依赖 Step 2 完成，因为需要 accepted_exts 来自注册表）
```

**四条独立路线可以并行进行**（但同一条线内必须顺序执行）：
- 路线 A：Step 1 → 2 → 3
- 路线 B：Step 4 → 5
- 路线 C：Step 6 → 7
- 路线 D：Step 1 → 8 → 9

## 回滚策略

每个 Step 开始前，确保 git 有一个干净的 commit。
- Step 1/4/6/8（纯新增）：直接 `git rm` 新文件即可回滚。
- Step 2/3/7（改现有文件）：`git checkout <file>` 还原。
- Step 5/9（最高风险）：`git revert` 或回到上一个 commit。

建议每完成一个 Step 就 `git commit -m "refactor: step N - 描述"`，保持细粒度提交记录。
