# PRD：文件列表与任务列表拆分

**版本**：V0.2.1  
**日期**：2026-03-30  
**状态**：待评审

---

## 1. 功能概述

当前版本（V0.1.1）的 `FileListWidget` 将"待转换文件"与"任务执行状态"混合在同一个列表中：每个文件项既承载文件元数据（文件名、大小），又内嵌任务状态（进度条、状态标签、task_id）。这导致文件生命周期与任务生命周期完全绑定，无法独立管理。

本功能将界面拆分为两个独立区域：

- **文件列表（左侧）**：用户管理待转换文件的暂存区，与任务执行解耦。
- **任务列表（右侧）**：展示已提交任务的执行状态与历史记录，独立管理。

---

## 2. 功能目标

| 编号 | 目标 |
|------|------|
| G-1 | 文件列表与任务列表在 UI 上左右分栏，职责清晰分离 |
| G-2 | 文件提交转换后从文件列表移除，任务条目进入任务列表 |
| G-3 | 任务完成后保留在任务列表作为历史，支持清除已完成 |
| G-4 | 失败任务支持一键重试，从任务列表直接发起 |
| G-5 | 批量图片合并任务（多图→一文件）在任务列表显示为一条总任务 |
| G-6 | 切换转换类型 Tab 时仅清空文件列表，不影响任务列表 |
| G-7 | 取消全部后任务显示为「已取消」状态，提供「重新开始」入口 |

---

## 3. 功能价值与业务意义

### 3.1 用户体验提升
- **减少操作混乱**：用户在查看任务进度时无需担心文件被误删，两侧列表各司其职，认知负担降低。
- **历史可追溯**：任务完成后不再消失，用户可回溯转换结果、定位失败原因。
- **高效重试**：失败任务无需重新拖拽文件，直接点击重试，减少重复操作。

### 3.2 架构合理性
- **关注点分离**：`FileListWidget` 回归纯粹的"文件暂存"角色，任务状态由新 `TaskListWidget` 独立管理。
- **生命周期解耦**：文件列表按用户意愿管理（手动添删），任务列表按任务生命周期自动管理（提交→执行→完成/失败）。
- **可扩展性**：未来支持任务队列持久化、任务日志导出等功能时，只需扩展任务列表模块。

## 4. 关键需求列表

### 4.1 必须实现（Must Have）

#### M-1：文件列表 - 提交后移除
- 点击「开始转换」后，被提交的文件从文件列表中移除。
- 若文件类型不匹配被跳过，则留在文件列表并给予提示（或直接拦截，不进入提交流程）。
- 批量图片合并任务：所有被提交的图片文件从文件列表移除。

#### M-2：任务列表 - 基础展示
- 每条任务项展示：文件名（或合并任务的合成标题）、转换类型、当前状态、进度条。
- 任务状态枚举：`等待中` / `转换中` / `已完成` / `失败` / `已取消`。
- 任务列表独立滚动，不随文件列表联动。

#### M-3：任务列表 - 历史保留
- 任务完成（成功/失败/取消）后，条目留在任务列表，不自动清除。
- 成功条目显示输出文件名或路径（可点击打开）。
- 失败条目显示简短错误信息。

#### M-4：任务列表 - 清除已完成
- 任务列表顶部操作栏提供「清除已完成」按钮。
- 点击后移除所有状态为"已完成"的任务条目，「已取消」和「失败」条目保留。

#### M-5：失败任务 - 一键重试
- 每条状态为「失败」的任务条目，显示「重试」按钮。
- 点击重试：使用原始参数（文件路径、转换类型、输出目录、选项）重新提交任务，条目状态重置为「等待中」。
- 重试时需校验源文件是否仍存在，不存在则提示错误。

#### M-6：批量图片合并任务 - 单条展示
- `images_to_pdf` / `images_to_word` 类型任务，无论包含多少张图片，在任务列表中显示为**一条**任务。
- 任务标题格式：`合并 N 张图片 → [输出文件名]`。
- 进度、状态、成功/失败信息均汇聚在这一条任务上。

#### M-7：UI 布局 - 左右分栏
- 主窗口内容区域采用左右分栏布局（`QSplitter` 或固定比例 `QHBoxLayout`）。
- 左侧：拖拽区 + 文件列表 + 文件操作按钮（选择文件、选择目录、清空列表）。
- 右侧：任务列表 + 任务操作按钮（清除已完成）。
- 建议初始比例左:右 = 1:1，支持用户拖动调整（使用 `QSplitter`）。
- 转换类型 Tab 和输出目录/开始转换按钮区域保持在顶部/底部全宽。

#### M-8：切换 Tab 时清空文件列表
- 切换转换类型 Tab 时，清空文件列表（现有行为是过滤不兼容文件，改为直接清空）。
- 任务列表不受影响。
- 若文件列表不为空，切换时弹出确认对话框："切换转换类型将清空文件列表，是否继续？"

#### M-9：取消全部 - 已取消状态保留
- 点击「取消全部」后，运行中和等待中的任务状态更新为「已取消」，条目保留在任务列表。
- 「已取消」条目显示「重新开始」按钮（区别于失败的「重试」，语义上更偏重用户主动取消后的恢复）。
- 「重新开始」逻辑与「重试」相同：使用原始参数重新提交，状态重置为「等待中」。

### 4.2 可选实现（Nice to Have）

#### N-1：任务列表 - 清除全部
- 增加「清除全部」按钮，一次性移除任务列表所有条目（运行中任务先取消）。

#### N-2：任务列表 - 成功条目点击跳转
- 点击已完成任务条目中的输出文件名，直接用系统默认程序打开文件（等同现有 `os.startfile`）。

#### N-3：文件列表 - 空状态提示
- 文件列表为空时，显示友好的空状态占位文字（如「拖拽文件到左侧区域」）。

#### N-4：任务列表 - 空状态提示
- 任务列表为空时，显示占位文字（如「暂无转换任务」）。

#### N-5：失败重试 - 批量重试
- 任务列表顶部添加「重试全部失败」按钮，批量重试所有状态为「失败」的任务。

## 5. 与现有模块的关联与依赖

### 5.1 需要新增的模块

| 模块 | 位置 | 说明 |
|------|------|------|
| `TaskListWidget` | `ui/widgets/task_list_widget.py` | 新增，替代 `FileListWidget` 中的任务状态职责 |
| `TaskListItem` | 同上或 `drop_zone.py` | 单条任务项 widget，含重试/重新开始按钮 |

### 5.2 需要修改的模块

| 模块 | 变更内容 |
|------|----------|
| `ui/widgets/file_list_widget.py` | 剥离所有任务相关属性和方法：`task_id`、`done`、`pending_paths`、`set_task_id()`、`mark_skipped()`、`reset_all_tasks()`、`update_task_state()`；`FileListItem` 移除进度条和状态标签 |
| `ui/widgets/drop_zone.py` | `FileListItem` 简化，移除 `status_label`、`progress_bar`、`set_waiting()`、`set_status()` 等任务状态相关逻辑 |
| `ui/main_window.py` | UI 布局改为左右分栏；任务回调（`_on_task_started/progress/finished`）改为调用 `TaskListWidget`；`_start_conversion` 提交后从文件列表移除文件并向任务列表添加条目；Tab 切换逻辑改为清空文件列表 |
| `core/task_manager.py` | 需在任务对象中保存原始参数（`input_path`、`conversion_type`、`output_dir`、`options`），供重试时读取；目前 `BatchImageTask` 已保存，`ConvertTask` 已保存，基本无需改动 |

### 5.3 数据流变化

**现在（V0.1.1）：**
```
文件拖入 → FileListWidget（文件+任务混合）→ 提交任务 → 状态回写同一个 widget
```

**目标（V0.2.1）：**
```
文件拖入 → FileListWidget（纯文件暂存）
    ↓ 点击开始转换
    文件移出 FileListWidget
    任务条目加入 TaskListWidget
    ↓ 任务执行
TaskManager 回调 → TaskListWidget 更新状态
    ↓ 失败/取消
TaskListWidget 显示重试/重新开始按钮
    ↓ 重试
TaskListWidget → MainWindow._retry_task() → TaskManager 重新提交
```

## 6. 潜在风险与注意事项

### R-1：重试时源文件已被删除
- 重试 / 重新开始前必须校验 `os.path.exists(input_path)`，文件不存在时提示用户并禁用按钮，避免任务提交后立刻失败产生误导。

### R-2：重试时输出目录已变更
- 重试使用的 `output_dir` 是任务创建时的快照值。若用户在重试前更改了输出目录，重试任务仍会输出到旧目录，需在 UI 上给予说明，或考虑重试时使用当前输出目录。

### R-3：批量图片任务的文件路径保存
- `BatchImageTask` 保存了完整的 `image_paths` 列表，重试时需要重新弹出 `getSaveFileName` 对话框询问输出文件名，不能直接复用旧的 `output_path`（旧文件可能已存在或被占用）。

### R-4：左右分栏在小窗口下的显示
- 最小窗口宽度 800px，左右各 400px，需验证两侧内容在最小宽度下不出现布局溢出。`QSplitter` 需设置合理的 `minimumWidth`，防止某一侧被压缩到不可用。

### R-5：切换 Tab 清空文件列表的用户感知
- 现有行为是"过滤不匹配文件"，改为"直接清空"会更激进。必须有确认弹窗（条件：文件列表非空时），防止用户误操作丢失已选文件。

### R-6：任务列表无限增长
- 长期使用下任务历史条目会积累，建议在「清除已完成」之外考虑 N-1（清除全部），或在任务数量超过阈值时给出提示。

### R-7：取消后「重新开始」与当前转换类型的一致性
- 用户取消任务后切换了 Tab（转换类型），点击「重新开始」仍使用旧的转换类型，这是预期行为（任务快照）。需要在任务条目上显示转换类型，避免用户困惑。
