## DocFlow 代码架构分析报告

---

### 一、功能总结

DocFlow 是一个 **本地离线文档格式转换工具**，基于 PySide6（Qt）构建桌面 GUI。

| 能力 | 技术依赖 |
|------|----------|
| Word/PPT → PDF | `win32com`（调用本机 Office/WPS COM 接口） |
| PDF → Word | `pdf2docx` |
| PDF → PPT | `PyMuPDF` + `python-pptx`（每页渲染为图片再插入） |
| PDF → 图片 | `PyMuPDF` |
| 图片 → PDF/Word | `Pillow` / `python-docx` |
| 多文件批量 | Qt `QThreadPool`（最多4线程并发） |
| 主题/设置持久化 | `QSettings` |

### 二、模块边界识别

```
main.py                     ← 入口点，仅启动 QApplication + MainWindow
│
├── ui/main_window.py       ← ⚠️ 超级大模块（见第三节）
├── ui/widgets/
│   ├── drop_zone.py        ← 拖拽区组件 + FileListItem 项组件
│   └── help_dialog.py      ← 帮助弹窗（独立，边界清晰）
│
├── core/task_manager.py    ← 任务调度（TaskManager + ConvertTask + BatchImageTask）
├── core/converter/
│   ├── office_converter.py ← COM 层：Word/PPT 转 PDF（有全局互斥锁）
│   ├── pdf_converter.py    ← PDF 转 Word/PPT/图片
│   └── image_converter.py  ← 图片合并为 PDF/Word
│
└── utils/
    ├── file_utils.py       ← 文件类型判断、路径生成
    └── logger.py           ← 日志初始化 + log_conversion 工具函数
```

**边界清晰的模块**：`office_converter`、`pdf_converter`、`image_converter`、`logger`、`help_dialog`，职责单一，对外暴露函数即可。

### 三、耦合点

#### 🔴 严重：main_window.py 承担了 5 种职责

这个文件（约 700+ 行）混合了：

| 职责 | 具体代码位置 |
|------|------|
| UI 布局构建 | `_init_ui()`、`_create_tab_page()` |
| 文件列表管理 | `_file_items` 字典、`_add_files()`、`_remove_file()`、`_clear_files()` |
| 转换类型路由 | `_get_current_conversion_type()`、`_get_accepted_extensions()` |
| 任务下发 & 结果处理 | `_start_conversion()`、`_on_task_started/finished()` |
| 主题 & 设置 | `_apply_theme()`、`_toggle_theme()`、`QSettings` |

这五件事全在同一个类里，互相依赖对方的状态。

#### 🟡 中等：扩展名规则定义了两遍

- file_utils.py 里有 `CONVERSION_MAP`（含 input/output 扩展名）
- main_window.py 里又自己写了一份 `_get_accepted_extensions()` 的 `ext_map` 字典

两者内容类似但不完全一致（`file_utils` 里有 `image_to_pdf`，`main_window` 里叫 `images_to_pdf`），日后如果新增格式，需要同步改两处。

#### 🟡 中等：`ConvertTask._do_convert()` 混了路由和调用

task_manager.py 里的 `_do_convert` 是一个长达 30 行的 `if-elif` 链，它既判断转换类型，又调用具体转换函数。这是**业务路由逻辑混入了调度层**。

同时，文件顶部已有模块级 `import`（`word_to_pdf`、`ppt_to_pdf`），方法内部又重复 `from core.converter.office_converter import ...`，存在冗余导入。

#### 🟡 中等：`_HINT_MAP` 是 UI 知识硬编码在类变量里

main_window.py 中 `_HINT_MAP` 字典把转换类型字符串（`'word_to_pdf'` 等）映射到提示文案。这份映射和 `CONVERSION_MAP`、`ext_map` 是三份并行的"转换类型知识"，任何类型改变都要同步三处。

### 四、潜在拆分点

这里只标出"如果这里乱了会很痛"的位置，不建议现在就动。

**拆分点 A｜文件列表管理器**
- 当前位置：`MainWindow._file_items` + `_add_files` + `_remove_file` + `_clear_files` + `_update_file_count`
- 可提取为：`ui/widgets/file_list_widget.py` 的独立 `QWidget` 子类
- 原因：文件列表逻辑（去重、目录展开、类型过滤）与 UI 布局无关，现在全埋在主窗口里

**拆分点 B｜转换类型注册表**
- 当前位置：`CONVERSION_MAP`（file_utils）、`ext_map`（main_window）、`_HINT_MAP`（main_window）
- 可合并为：一个单一的 `CONVERSION_REGISTRY` 结构，每个条目包含 `input_exts`、`output_ext`、`hint_text`
- 原因：三份数据描述同一个领域概念，新增转换类型时要改三处

**拆分点 C｜`ConvertTask._do_convert()` 的路由逻辑**
- 当前位置：task_manager.py 里的 `if-elif` 链
- 可提取为：一个独立的 `dispatch(conversion_type, input_path, output_dir, options)` 函数放在 core 下
- 原因：调度层（QRunnable）不应该知道"哪种类型调哪个函数"，这是业务逻辑

**拆分点 D｜主题管理**
- 当前位置：`MainWindow._apply_theme()`、`_toggle_theme()`、`QSettings` 操作散落在 `__init__` 和方法里
- 可提取为：`ui/theme_manager.py`（或简单的 `ThemeManager` 类）
- 原因：主题与界面逻辑无关，未来主题变多时会更难维护

### 五、风险提示

> office_converter.py 里用 `threading.Lock` + `pythoncom.CoInitialize/CoUninitialize` 保护 COM 调用，但 `QThreadPool` 的线程不是主线程——COM 串行锁是正确的，但如果 `CoUninitialize` 在异常路径下执行顺序有误，可能导致后续任务的 `CoInitialize` 失效。目前 `finally` 块处理是正确的，只是需要注意不要在锁外调用 COM 对象。
